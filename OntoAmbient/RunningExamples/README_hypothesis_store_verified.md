# hypothesis_store_verified.json

## Purpose

This file is `hypothesis_store.json` **updated in-place with CoVe verification results**. Each PENDING hypothesis now has a final verdict (`VALIDATED`, `REFINED`, or `REFUTED`), a confidence score, CoVe verification questions and answers, a tool call plan, and the full evidence bundle statistics.

This is the **primary output of Phase B (Hypothesis Verification)** — the most computationally expensive phase, requiring Planner + Executor + CoVe for each of the 35 hypotheses.

---

## File Format

Same JSON array structure as `hypothesis_store.json`. The top-level generation records are unchanged; only the **hypothesis objects within each record** are updated with 9 new fields.

```json
[
  {
    "finding_id": "F07",
    "run_id": 1,
    /* ... all original generation record fields unchanged ... */
    "hypotheses": [
      {
        /* All original Phase 4 fields */ +
        /* 9 new Phase 5 verification fields */
      },
      ...
    ]
  }
]
```

---

## New Fields Added to Each Hypothesis Object

These 9 fields are added to each hypothesis object by the Planner → Executor → Consolidator pipeline:

| Field | Type | Description |
|-------|------|-------------|
| `status` | string | **Updated from `PENDING` to:** `VALIDATED`, `REFINED`, or `REFUTED` |
| `confidence_score` | float | CoVe-derived confidence (0–1). Replaces the generation-time `plausibility_score` as the primary quality signal |
| `flaws_found` | int | Number of CoVe verification questions (0–4) that identified a genuine flaw in the hypothesis |
| `final_statement` | string | Refined hypothesis statement produced by CoVe when verdict is `REFINED`. For `VALIDATED` and `REFUTED`, this is the original statement |
| `cove_questions` | list[dict] | The 4 verification questions generated by the Consolidator with their answers and flaw assessments (see sub-schema below) |
| `evidence_coverage` | string | Plain-English summary of evidence completeness from the Consolidator |
| `baseline_interpretation` | string | Consolidator's interpretation of what the evidence shows before verdict assignment |
| `evidence_steps` | int | Number of tool call steps executed by the Executor (typically 5) |
| `evidence_rows` | int | Total rows returned across all evidence steps |
| `tool_plan` | list[dict] | Full ordered tool call plan from the Planner (see sub-schema below) |
| `verification_time_s` | float | Wall-clock seconds for Planner + Executor + CoVe combined |

## Verdict Assignment Logic

| Verdict | `flaws_found` | Evidence Condition | Meaning |
|---------|---------------|--------------------|---------|
| `VALIDATED` | 0–1 | Evidence positively supports the causal chain | Strong support — appropriate for direct action |
| `REFINED` | 2 **or** any | Evidence incomplete (source returned 0 rows) | Plausible but unconfirmable — high-value investigative lead |
| `REFUTED` | 3–4 | Evidence **actively contradicts** the hypothesis | Evidence contradicts the claim — prevents false positive propagation |

> **Critical rule:** Missing data (`tool returned 0 rows`) = `REFINED`, **not** `REFUTED`. `REFUTED` is reserved exclusively for cases where retrieved evidence contradicts the hypothesis (e.g., all accused providers have `FraudLabel=0`, or effect direction is reversed in data).

## `cove_questions` Sub-Object Schema

Each item in `cove_questions` covers one of four verification dimensions:

```json
{
  "question": "The specific verification question asked",
  "dimension": "temporal | confound | completeness | statistical",
  "answer": "The answer derived from the evidence bundle",
  "flaw_detected": true or false,
  "flaw_description": "Explanation of the flaw if detected (empty string if no flaw)"
}
```

| Dimension | What It Checks |
|-----------|---------------|
| `temporal` | Does the evidence timeframe align with the period covered by the finding? |
| `confound` | Could an alternative innocent explanation produce the same evidence pattern? |
| `completeness` | Does the retrieved evidence cover all aspects declared in `evidence_needed_to_verify`? |
| `statistical` | Is the statistical pattern in retrieved evidence consistent with the claimed effect direction and magnitude? |

## `tool_plan` Sub-Object Schema

```json
[
  {
    "step": 1,
    "tool": "query_provider_db",
    "purpose": "What this step is fetching and why",
    "parameters": { "...filter params..." },
    "uses_output_of": null
  },
  {
    "step": 2,
    "tool": "query_claims_db",
    "purpose": "...",
    "parameters": { "...": "..." },
    "uses_output_of": 1
  },
  ...
  {
    "step": 5,
    "tool": "query_audit_store",
    "purpose": "Verify ground truth finding verdict from Phase 2b",
    "parameters": { "finding_type": "PeerGroupReimbursementDeviation" },
    "uses_output_of": null
  }
]
```

`uses_output_of` specifies which prior step's output provides filter parameters for this step (enabling automatic key injection by the Executor). `null` means this step has no dependency.

## What the 3 REFUTED Hypotheses Represent

The REFUTED hypotheses are the most diagnostically valuable outputs — they demonstrate CoVe catching domain-intuitive hypotheses that the data contradicts:

1. **Upcoding + low-chronic-condition patients (F05):** CoVe detected that retrieved beneficiary data shows fraud providers' patients have *more* chronic conditions than expected (d = −0.030, opposite direction). The hypothesis that fraud providers preferentially target low-complexity patients is REFUTED.

2. **Phantom billing via zero-duration claims (F09):** CoVe detected that zero-duration claims show a *lower* fraud rate in evidence than non-zero claims (d = −0.171, opposite direction). The hypothesis that zero-duration claims are a primary phantom billing mechanism is REFUTED.

3. **Physician network hub billing fraud (F03-variant):** One physician network hypothesis was REFUTED when evidence showed the network's FraudAssociationRate was below the AX3 threshold in the retrieved physician records.

Without CoVe, all 35 initially plausible hypotheses would have propagated to stakeholders. CoVe correctly catches 3 false positives (8.6%), while appropriately expressing uncertainty about the remaining 32 (REFINED) rather than over-claiming VALIDATED status from incomplete evidence.
